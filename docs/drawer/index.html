Drawer

Drawer Options

  • Whether hamburger is link or button.
  • Whether focus is trapped in drawer.
  • Position and number of close buttons:
    • One before menu links
    • One before menu links and one after menu links
    • One after menu links
  • How much to hide outside of drawer:
    • Everything
    • Main Content
    • Everything below header
  • Where to set focus on clicking hamburger:
    • Leave on hamburger
    • Set focus to close button in menu group
    • Set focus to entire menu
    • Set focus to first link in menu
  • Whether to collapse menu when focus leaves menu.
  • Whether pressing Escape collapses menu.

Considerations

  • Making hamburger a link (pointing to the heading of the menu) might conform better to progressive enhancement. However, with JS enabled, you'd want the Menu to be a button, since it is really performing an action. Since you aren't supposed to change an element's role, you'd want to destroy the link and create the button. This is starting to get complicated.
  • NVDA only announces labels on ARIA landmarks, not announce groups or articles.
  • Some screen readers (NVDA?) may not announce changes to the aria-expanded property (though I think they will announce its state when user moves to button. Therefore, the text of the button should probably change, but that may present confusing cues, if read together with the aria-expanded state change on SRs that support that support announcing that property change. Button text should be chosen carefully for this reason. NOTE: Is it even a good thing for this change in state to be announced. Since users are clicking the button, they should expect that it will expand the menu, and may not want to be bothered by this extra noise.
  • Consider whether clicking on Search button causes the same behavior.
