So, I've given you some sense of how somebody who uses a screen reader might orient themselves in a table, but perhaps you're wondering how the screen reading software knew to give me all those useful cues.

In this case, it knew because I had used the tools available to me in Canvas's rich content editor to make the table and to indicate what cells I wanted to serve as row and column headers.  

Now, that may seem obvious, but let me show you an example of a way I could have presented the information that wouldn't have given screen reader users those same cues.

28

Click and drag to select text from the start of the "table" to about halfway through.

 The page https://canvas.vt.edu/courses/69527/pages/table-headers-non-example-1/edit is shown in the Safari browser.

Here we have what looks like a table, but I haven't indicated programmatically that it is a table in any way that a screen reader can hook into.  Instead, I've just aligned things visibly with whitespace by repeatedly pressing the spacebar key, and I have added the horizontal bar character to simulate cell borders.  I can show you by deleting a few characters.  

29

Place the cursor in the table after the horizontal bar following the word MAGNIFICATION and press the Delete key a few times.

  

See?

30

    

Unfortunately, doing things this way means that a screen reader user won't be told they have entered a table when they encounter it, nor will they be told how many columns and rows are in the table.  They won't be able to move between cells, either, but, instead, will have to read things line by line, in order. They will also be given no cue indicating where they are in this supposed table as content is read to them.  It's just a horrible user experience.

To make a real table that provides the right cues and affordances to screen reader users, you need to use the table functionality that Canvas's rich content editor provides for you.

To create a real table in Canvas you click the Table button menu in the editor's toolbar, and in the Table menu option, you just choose the number of rows and columns you want and, presto, you've got yourself a table.

When a screen reader user encounters this version of the table, they will be told they are in a table and what its name is, and also how many rows and columns are in the table.  They will even be able to navigate vertically and horizontally between cells instead of having to read through it line by line in order.  

However, even though this table looks just like the first one I demoed for you, there is one important difference.  

In this version of the table, if I use a screen reader to navigate from cell to cell, it won't tell me the header name whenever I enter a new column or row.  If you'll recall, this was very useful information in helping me find the information I was looking for.  

Why won't it announce the header name?  It sure looks like there are row and column headers.  But looks can be deceiving. In this table, I only changed the appearance of the cells that were to serve as headers.
